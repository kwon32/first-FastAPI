# 4인 개발팀을 위한 Git 협업 가이드 (FE 2명 + BE 2명)

프론트엔드 2명, 백엔드 2명으로 구성된 소규모 팀을 위한 가장 효율적인 Git 협업 전략(**GitHub Flow**)입니다.

---

## 1. 저장소(Repository) 구성 전략

충돌 방지와 배포 편의성을 위해 **저장소를 분리**하는 것을 강력 추천합니다.

*   📂 **Frontend Repository**: `team-project-fe` (React, Vue 등)
*   📂 **Backend Repository**: `team-project-be` (Spring, Django, FastAPI 등)

---

## 2. 브랜치 전략: GitHub Flow

복잡한 규칙 없이 **속도**와 **안전**을 챙기는 방식입니다.

### 브랜치 종류
1.  **`main`**: 언제든 배포 가능한 **최종본**. (절대 직접 수정 금지 🚫)
2.  **`feat/...`**: 각자 기능을 개발하는 **작업 공간**.

### 작업 순서 (Workflow)
1.  **[최신화]**: 작업 시작 전, 항상 `main`을 당겨옵니다.
    ```bash
    git checkout main
    git pull origin main
    ```
2.  **[브랜치 생성]**: 기능별로 브랜치를 만듭니다.
    ```bash
    git checkout -b feat/로그인기능
    ```
    *   *Naming Rule 예시*: `feat/login-ui` (FE), `api/login-logic` (BE)
3.  **[개발 & 커밋]**: 열심히 코딩하고 커밋합니다.
4.  **[PR 생성]**: GitHub 웹사이트에서 **Pull Request(PR)**를 보냅니다.
    *   *의미*: "내 작업이 끝났으니 `main`에 합쳐주세요!"
5.  **[코드 리뷰]**: 팀원(동료)에게 리뷰를 받습니다. (오타, 로직 확인)
6.  **[Merge]**: 리뷰가 통과되면 `main` 브랜치로 합칩니다.

---

## 3. FE & BE 협업 포인트 (중요!)

서로 기다리지 않고 병렬로 작업하기 위한 핵심 규칙입니다.

1.  **API 명세서 선행 (계약서 작성)** 📝
    *   코딩 전에 모여서 URL, 요청/응답 JSON 형태를 먼저 정합니다.
    *   *Tool*: Swagger, Notion, Excel 등
2.  **Mocking (가짜 데이터 활용)** 🎭
    *   **FE**: 합의된 JSON 형태의 가짜 데이터를 만들어 화면을 먼저 개발합니다.
    *   **BE**: 합의된 명세에 맞춰 로직을 구현합니다.
3.  **도킹 (연동)** 🚀
    *   BE 개발이 끝나 `main`에 합쳐지면, FE는 가짜 데이터를 실제 API 주소로 교체합니다.

---

## 4. 우리 팀의 Ground Rule (약속)

1.  **1일 1PR (작게 쪼개기)** ✂️
    *   거대한 코드는 리뷰하기 힘듭니다. 기능 단위로 잘게 쪼개서 자주 합치세요.
2.  **출근하자마자 `git pull`** ☀️
    *   남이 작업한 최신 코드를 내 컴퓨터로 가져와야 충돌을 막을 수 있습니다.
3.  **친절한 PR 메시지** 💌
    *   제목: `[FE] 로그인 페이지 UI 구현`
    *   내용: 구현 내용 설명 + **스크린샷 첨부** (스크린샷이 있으면 리뷰 속도가 2배 빨라집니다!)
